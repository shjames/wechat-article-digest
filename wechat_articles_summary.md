### 1. 这下，微信终于忍不住了

**文章总结：**

**核心摘要**

微信推出“小龙虾”（OpenClaw插件）的真正原因，是其护城河——超过13亿用户构建的网络效应——正在受到挑战。随着AI时代到来，大量用户开始将时间花在Telegram、WhatsApp等更适配AI的平台，导致微信的“入口”地位被动摇。文章指出，微信目前只做了“接口”层面（保住入口），而非更聪明的大模型、更丰富的技能或更强大的操作系统。对普通人而言，机会在于开发各类垂直“Skills”技能，而非与大厂竞争接口或大模型。

**关键信息**

1. **微信推出AI插件**：3月22日微信正式推出ClawBot插件，支持远程操作设备、处理文件、管理日程等，但功能较为基础（不能拉群、纯文本交互、处理复杂任务费劲）。

2. **微信的护城河**：13亿+用户及由此产生的网络效应，使微信成为时代的“连接”基础设施。微信支付、游戏分发等业务的成功都依赖于此。

3. **护城河受挑战的原因**：用户为使用AI工具（如“小龙虾”），开始登录Telegram等平台并花时间在这些平台，甚至顺便与朋友交流，导致社交活跃用户分流。

4. **“小龙虾”的四层模型**：①大模型（Model）②技能（Skills）③智能体操作系统（AgentOS）④接口（Interface）。前三者微信可不管，但“接口”是其护城河，必须守住。

5. **微信的谨慎策略**：“微信龙虾”更像一个“特区”，与微信主生态严格隔离（文章、图片、聊天记录不能转发）。

6. **普通人机会**：不在大模型、不在操作系统、不在接口（这些是巨头战场），而在**Skills技能开发**——如PPT生成、法律咨询、旅游定制路线等技能。未来可能出现类似App Store的Skills市场。

7. **文章性质**：这是一篇**观点/分析文章**（刘润公众号），分析AI时代微信的战略选择和普通人的机会。

8. **广告信息**：刘润将于5月16日在上海举办“年中大课”，主题为“落地”，分享AI技术落地案例。

**核心摘要**

刘润在这篇观点文章中，对当前火热的“养龙虾”（OpenClaw智能体）现象泼冷水。他指出，龙虾本质是一个效率工具，类似“聪明的实习生”，能做你交代的事，但无法帮你找到更值钱的问题。大多数人每天忙的那些事本身并不贵，用龙虾提高效率后还是赚那么多钱。文章提醒：龙虾是放大器，能让你做事更快，但快不等于赚更多；面对这波热潮，应重点思考自己有没有值得被放大解决的问题，而非盲目跟风安装。

**关键信息**

- **“龙虾”是什么**：OpenClaw是一个开源智能体，因图标是龙虾，被戏称为“养龙虾”。它能操控电脑、接入聊天软件、有长期记忆、可自动执行重复任务。
- **热潮现状**：淘宝89元远程部署服务上千人下单；腾讯免费安装当天上千人排队。
- **龙虾的核心局限**：本质是“实习生级执行”，只能帮人把交代的事做得更快，无法帮人发现未被满足的需求、找到更值得做的事、或建立壁垒。
- **作者的核心论点**：赚钱的起点不是效率高低，而是手里有没有“值钱的问题”。龙虾解决效率问题，但整理资料、发邮件这类工作本身就不贵，做完还是原来那么多钱。
- **这波龙虾热的本质**：焦虑的集体释放，类似去年的DeepSeek热、元宇宙、Web3热，最终赚大钱的是“卖铲子”的人（帮人安装配置、教怎么用）。
- **三条建议**：
  1. 别装在主力电脑上，工信部已预警信息泄露风险；
  2. 考虑清楚费用，Token消耗累计可能不亚于招实习生；
  3. 搞清楚自己有没有值钱的问题——找不到，龙虾只会让你更焦虑。

**文章性质**

这是一篇**观点评论文章**，作者刘润对当前技术热潮表达理性分析和建议，属于“泼冷水”类型的行业观察。

- 资源利用率：传统数据库架构僵化、静态分配导致资源“空转”；分布式数据库通过动态调度提升利用率

- 传统架构组件众多（数十个独立模块），硬件资源分散、管理开销大
- 采用静态资源分配，业务低谷期大量服务器闲置，高峰期扩展粒度粗放
- 数据冗余度高，通常需3副本甚至更多
- 单体式设计紧耦合，故障隔离能力弱，容灾备份需额外服务器投入
- **结论**：传统分库分表架构比原生分布式架构多消耗约60%服务器资源

- **负载均衡三大机制**：
  - 自动分片：基于一致性哈希与范围分区，避免“热点”分片
  - 热点迁移：动态将高负载数据片迁移至空闲节点，无需停机
  - 副本均衡：智能分布全功能、日志、只读副本，提升资源利用
- **高压缩能力**：列式存储+编码优化+自研压缩算法，压缩比3:1-5:1
- **多租户架构**：资源池化+逻辑隔离，资源利用率从20%-30%提升至60%以上

### 4. 什么时候 Vibe Coding（与 AI 结对编码），什么时候 Harness Engineering（设计编码环境，AI 自主完成全部编码）？

当你设计了完整编码环境：

「开发（SDD + TDD）、验证、提PR、Review 代码、基于 Review 修改、继续提 PR、PR 通过合并」

这套 编码Loop 完全由 Agent 自主完成。那么这类编码工作就是 Harness Engineering。

接下来，假设你想修改下环境，比如：
你使用 superpowers 串联 SDD 完整流程，superpowers 默认为每个任务创建一个 git worktree，虽然能保证代码隔离，但跑测试时你希望有全新的 DB 副本，所以希望将流程改成：

「每个任务创建一个 Dev Container（Docker 环境），有全新的数据副本与代码，跑完任务提交 PR 通过后再销毁环境」

那就需要修改 superpowers 内的 using-git-worktree SKILL。  

如果是修改代码，你有完整的 Loop 可以让 Agent 执行。但修改 Skill 内的自然语言，你还没设计一套完整的 Loop。所以你选择：
1. 告诉 Agent 你希望如何修改 using-git-worktree SKILL
2. Agent 修改完后你再人工 Review
3. Review 没问题后跑几下试试效果

这个过程就是 Vibe Coding。

核心区别就是：不管是针对 编程语言还是自然语言，关键是「你有没有一个完整的环境，可以对 Agent 的行为进行约束、观测、校验、回退」

有，就是 Harness Engineering
没有，需要人工介入，就是 Vibe Coding

- 发布时间: 2026-03-24 11:32:20
- 原文链接: https://mp.weixin.qq.com/s/mT1TWMjGrBXw8n_-IuQqlA

**文章总结：**

## 核心摘要

文章介绍了AI辅助编程的两种模式——**Vibe Coding**（与AI结对编码）和**Harness Engineering**（设计编码环境让AI自主完成）。两者的核心区别在于：是否有完整的环境可以对Agent的行为进行**约束、观测、校验、回退**。有完整闭环是Harness Engineering，需要人工介入则是Vibe Coding。

## 关键信息

1. **Harness Engineering定义**：设计完整的编码环境（如「开发→验证→提PR→Review→修改→合并」闭环），Agent可自主完成全部编码工作，无需人工介入。

2. **Vibe Coding定义**：当需要修改环境配置（如SKILL内的自然语言描述）时，由于没有完整的自动化闭环，需要人工告知Agent修改方向、人工Review、人工测试验证，这个过程就是Vibe Coding。

3. **核心判断标准**：能否对Agent行为进行约束、观测、校验、回退——不区分编程语言还是自然语言，关键看是否有完整环境支撑。

4. **实践案例**：使用superpowers串联SDD流程时，若要将git worktree改为Dev Container方案，需要修改SKILL配置，此时属于Vibe Coding范畴。

**关键信息**

1. **RNI（推荐摄入量）**：建议的充足摄入量，可保证绝大多数个体摄入充足，通常作为主要参考标准。

2. **AI（适宜摄入量）**：在缺乏充分数据时用于估计群体数据，并非必须达到，该数值往往比RNI偏高。

3. **PI（预防非传染性慢性病的建议摄入量）**：针对一些容易摄入不足的营养素，达到此量对健康更有益处。

4. **实例说明**：
   - 钾：RNI为2000mg，PI为3600mg
   - 7-12月龄婴儿钠的AI为350mg
